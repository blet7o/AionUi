## 2025-03-17 - [CRITICAL] Fix Timing Attack & Auth Bypass in constantTimeVerify
**Vulnerability:** The `constantTimeVerify` function in `AuthService.ts` was vulnerable to timing attacks when user was not found, as passing `'dummy'` to `bcrypt.compare` fast-failed in ~0.1ms compared to ~100ms for a real hash. Furthermore, a severe string-padding logic bug in the non-hashed code path `crypto.timingSafeEqual` caused it to return `true` when inputs were prefixes with padding (e.g. `'admin0'` and `'admin'`).
**Learning:** `bcrypt.compare` fails extremely quickly on invalid hash strings, nullifying attempts to mitigate timing attacks if a dummy non-hash string is passed. Manual string length normalization using `.padEnd` with a fixed character (`0`) creates collisions causing valid bypasses since both padded strings end up being identical.
**Prevention:** When simulating password hashing for timing attack prevention, you must use a real, pre-computed bcrypt hash string format. When doing constant-time string comparisons, do not pad strings; check lengths in a constant-time way, or use double HMAC, or fall back to comparing the buffer against itself.
